## Basics of Kubernetes

With Kubernetes, Nolly becomes :

```
Agent → submit_scan_tool → create Kubernetes Job → runs Nmap container → finishes → dies
```

* Basically an automated container runner & manager

#### Core Concepts

| Term           | Meaning                                        |
| -------------- | ---------------------------------------------- |
| **Pod**        | A running container (your Nmap scan)           |
| **Job**        | “Run this once and finish” (perfect for scans) |
| **Deployment** | Long-running apps (your API)                   |
| **Cluster**    | The whole Kubernetes system                    |
| **kubectl**    | CLI to control everything                      |

##### Commands :

* **Check cluster :** 
  
  ```bash
  kubectl get nodes
  ```

* **See Running things (pods) :**
  
  ```bash
  kubectl get pods
  ```

* **See jobs :**
  
  ```bash
  kubectl get jobs
  ```
  
  * **See job details :**
    
    ```bash
    kubectl describe job <job_name>
    ```
  
  * **Check logs :**
    
    ```bash
    kubectl logs job/<job_name>
    ```
  
  * **Applying a job :**
    
    ```bash
    kubectl apply -f job.yaml
    ```
  
  * **Delete job (cleanup) :**
    
    ```bash
    kubectl delete job <job_name>
    ```
  
  * **Watch live Jobs** (IMPORTANT) :
    
    ```bash
    kubectl get pods -w
    ```

> **Example :** Defining a job & running it
> 
> ```yaml
> apiVersion: batch/v1
> kind: Job
> metadata:
>   name: nolly-scan
> spec:
>   template:
>     spec:
>       containers:
>       - name: scanner
>         image: instrumentisto/nmap
>         args: ["-sV", "target-web"]
>       restartPolicy: Never
> ```
> 
> **Run via :** `kubectl apply -f job.yaml`
> 
> **Kubernetes does :** `create pod → run nmap → finish → store logs → mark job complete`

## Basics of Kind

> **Kind** $\rightarrow$ Kubernetes In Docker
> 
> "**Run a Kubernetes cluster using Docker containers**"

* tool that creates a local Kubernetes cluster on our machine using Docker
1. Must create a kind, I created the `n0lly` kind
   
   ```bash
   kind create cluster --name n0lly
   ```

2. Check the clusters :
   
   ```bash
   kind get clusters
   ```

3. Check namespace
   
   > `Namespace` : logical container inside Kubernetes to group resources
   > 
   > * "folders inside our systems"
   
   ```bash
   kubectl get ns
   ```

4. Check which cluster `kubectl` is trying to use
   
   ```bash
   kubectl config current-context
   ```

### Change the lab test for k8s

Added : `lab/k8s-target-web.yaml` :

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: target-web
  namespace: nolly
spec:
  replicas: 1
  selector:
    matchLabels:
      app: target-web
  template:
    metadata:
      labels:
        app: target-web
    spec:
      containers:
        - name: web
          image: nginx:alpine
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: target-web
  namespace: nolly
spec:
  selector:
    app: target-web
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
```

pull alpine : `docker pull nginx:alpine`

load to `n0lly` kind : `kind load docker-image nginx:alpine --name n0lly`

Verify : `docker exec -it n0lly-control-plane crictl images | grep nginx`

**Apply :** `kubectl apply -f k8s-target-web.yaml`

**Check :**

```bash
kubectl -n nolly get pods
kubectl -n nolly get svc
```
